Hi @Krylva! I don't use the mailer integration but from my point of view, there are many places to check before further investigations of your problem. The short checklist may look like:

  1. Have you installed the mailer integration correctly?
  2. Where is your config file placed? By default, the mailer tries to load configuration from the ~/.alerta.conf file. So you should be sure that a user under which the mailer runs can find this config file in its home folder. If you have set the ALERTA_CONF_FILE environment variable, the config file should be placed by a path from this variable.
  3. How do you run the mailer? As I can see it is supposed to be run manually like this:
    export SMTP_PASSWORD=okvqhitqomebufyv
alerta-mailer
    Do you run it in this way or use systemd or something?
  4. How you check logs? They should be available only in a console as an output of the mailer command but I'm not sure about this now.
  5. Do such logs contains something interesting?
  6. Try to enable the debug mode by setting the DEBUG environment variable:
    export DEBUG=True
alerta-mailer

A simple hack to check that the mailer tries to use a correct config file: put bad content (maybe, any binary data) to your config file and try to start the mailer then. It should fail with an error like this: Problem reading configuration file %s - is this an ini file?

P. S. I don't have any experience with the mailer so all those thoughts were got from the source code of the mailer.

Dec 17 2019 07:13 UTC

@jgoetz You don't need to activate the virtual environment because the next command uses pip directly from the created virtual environment:

alerta/bin/pip install --upgrade pip wheel alerta-server alerta uwsgi

The same for the SystemD service: it uses uwsgi directly from the created virtual environment:

[Service]
ExecStart=/opt/alerta/bin/uwsgi --ini /etc/uwsgi.ini
@jgoetz but if you want to perform some manual actions with the Alerta installation, it may be very useful to activate the virtual environment.
