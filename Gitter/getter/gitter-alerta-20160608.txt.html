morning. I'm looking at using alerta to send various notifications for users (chat/email/etc). there is one problem I can't see the obvious solution for - I have to send a repeated notifications. Ie for example to send a notification if critical alert appears and repeat a notification (possibly even with changed settings) if 30min later this critical alert is still there.
my backends send regular updates quite often, so I can hook to post_receive(), but AFAICS I need some more fields into alert data - at least for a timestamp for a last notification sent.
does it make sense? IMHO it could be a generic feature, no?

hi @hasso you could do this one of two ways.
1/ as you suggest, add an attribute to each alert that keeps track of when the last notification for that alert was sent. this can be easily achieved using a custom attribute. see http://docs.alerta.io/en/latest/design.html?highlight=custom#tags-and-custom-attributes and http://docs.alerta.io/en/latest/api/alert.html#attributes
2/ using a long-running process to keep track of which alerts have been notified and when. some examples of this type of integration can be found here https://github.com/alerta/alerta-contrib/tree/master/integrations though none are exactly what you are after.
the second option can make use of an AMQP message queue to send alerts to the long-running daemon that is listening for new alerts and takes actions accordingly, like looking up a table to see when the last time that env-resource-event combination was last notified.
let me know if you have any more questions or i can help you further.
