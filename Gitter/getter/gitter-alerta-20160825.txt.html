Can you tell what was the reason to evolve ui into client side templating, instead of server side?
Nick Satterly
@satterly
Aug 25 2016 08:06 UTC
I started off using flask Jinja templates but I found it a struggle when the UI started to get more complex. New functionality was being added by bolting on JavaScript bits and pieces and it wasn't very flexible. I decided to split the project into api and ui repos so I could independently develop both. I then decided to use angularjs to build a new ui which is where we are now. There are always pros and cons with different approaches. From a security point of view (auth and CORS for example) its much easier to have a server side UI, I think. Though I think client side JavaScript is more flexible not just for development but deployment as well. Eg the alerta web UI can be served as static assets from a cdn.
It also means anyone could develop a UI that worked with the api as long as the api was well documented.
