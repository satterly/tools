Hello together

I try to deploy Alerta on Kubernetes with his own configuration files.
For that I have create a Config-Map with the "alertad.conf" and "config.json"
In my case I have mounted this Config-Map to "/alerta-config/".
So far so fine.
If I start Alerta I can see the config files.

ls -l /alerta-config/
total 0
lrwxrwxrwx. 1 root root 18 Oct 3 07:48 alerta.conf -> ..data/alerta.conf
lrwxrwxrwx. 1 root root 18 Oct 3 07:48 config.json -> ..data/config.json

But I don't know what I need in my Kubernetes Deployment YAML so that Alerta
is using these config files.

I have tryed with this... But I see it is wrong:
args: ["-v /alerta-config/alertad.conf:/app/alertad.conf", "-v alerta-config/config.json:/web/config.json"]

Thanks in Advanve
Manfred

Nick Satterly
@satterly
Oct 03 2019 18:58 UTC
@iDemonix The “AngularJS” web UI is end of life and will not be updated again. The VueJS web UI will be maintained going forward. This is the repo for the VueJS web UI https://github.com/alerta/alerta-webui
Is that what you needed to know?
@joaorebelo-ar I’ve created a branch where it is possible to set the max latency at the server. See alerta/alerta@897f7fd
I’m considering if this is the best way forward or there are better alternatives.
The CLI would get the heartbeat status from the server response. It would not be configurable by the CLI. ie. a CLI wanting to list stale/slow heartbeats would not be able to configure the threshold for max latency. that would be determined by the server as a global setting. Your opinion on this approach would be useful. Thanks.
Nick Satterly
@satterly
Oct 03 2019 19:05 UTC
@ngohoa211 Yes, the Customer field is used to namespace alerts by customer. ie. if two different customers have an alert with env=prod, resource=foo, event=bar then they should not correlate or deduplicate each other. Allowing users to arbitrarily change the customer field would break alerta in a bad way.
Thinking about it again, I don’t think just sending the same alert again with different customer is going to work.
It is not the intention that users would assign alerts to different customers as they arrive. The customer field should be automatically assigned. You could easily use a plugin to do this based on either information in the alert or by querying a config management database or whatever.
Nick Satterly
@satterly
Oct 03 2019 19:11 UTC
@ManCon There’s an integration for Alerta and Opsweekly here https://github.com/alerta/alerta-contrib/tree/master/integrations/opsweekly
If it’s not updated anymore I’m not sure what the alternative is. Writing your own reporting scripts is the only thing I can suggest right now.
There’s an alerta Helm chart here that might help https://github.com/alerta/docker-alerta/tree/master/contrib/kubernetes/helm/alerta
It’s for the old version of the web UI (the angularJS one) so you’ll need to change that. I’ll update some time soon, I hope.
