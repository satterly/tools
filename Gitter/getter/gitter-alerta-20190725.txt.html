Nick Satterly
@satterly
Jul 25 2019 08:01 UTC
@rdxsuper if NOTIFICATION_BLACKOUT is True then the alert is accepted but no plugins after the blackout plugin will be triggered. If NOTIFICATION_BLACKOUT is False then the alert will be rejected and not processed at all.
@Krylva To fix the status “resolved” problem set DEFAULT_NORMAL_SEVERITY=‘ok’ in your alertad.conf server config file.
For the problem where alerts are visible in the web UI, can you see any alerts in the web UI or is it just Prometheus webhook alerts?

Nick thank you for your attention but i noticed the same issue that described above. Problem was caused absence of normal severity level in /etc/alerta.conf. Sorry for bother you. Problem is solved afeter some hours of investigation. And I will add the default severity level in config.

Nick Satterly

Jul 25 2019 09:03 UTC

@Krylva I’ve updated the docs to reflect this change …

The default “normal” severity is normal however it is no longer a valid severity following the changes made above so you must explicity set the default severity to ok by adding:

DEFAULT_NORMAL_SEVERITY = ‘ok'

see https://docs.alerta.io/en/latest/gettingstarted/tutorial-1-deploy-alerta.html#tutorial-1

Nick Satterly
@satterly
Jul 25 2019 17:27 UTC
@rdxsuper plugins are run in the order in which they are defined, left to right.
So if you put blackout last in the list it won't have any effect of what you want it to do is suppress downstream notifications. It should be listed as early as possible, usually straight after the reject plugin.
@teravik it's probably a CORS issue. Have to added xx.xx.xx.xx to the list of CORS_ORIGINS in the server config? (whatever xx.xx.xx.xx is)
Nick Satterly
@satterly
Jul 25 2019 17:33 UTC
